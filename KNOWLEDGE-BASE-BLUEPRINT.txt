================================================================================
  KNOWLEDGE BASE BLUEPRINT — Prompt Generico per la Costruzione di una KB
  Architettura basata su MkDocs + Material Theme
================================================================================

Questo prompt descrive in modo completo l'architettura di una knowledge base
scalabile, modulare e moderna. Forniscilo integralmente a qualsiasi AI assistant
(ChatGPT, Perplexity, Claude, Gemini, ecc.) per generare tutti i file necessari.

Il carattere [PLACEHOLDER] indica un campo che DEVI compilare prima di usare
il prompt. La sezione 0 è quella da compilare.


================================================================================
  SEZIONE 0 — DA COMPILARE PRIMA DI USARE IL PROMPT
  (Rimuovi queste istruzioni e lascia solo i valori compilati)
================================================================================

NOME PROGETTO:
  [Inserisci il nome del progetto — es. "Data Engineering Knowledge Base",
   "Cybersecurity Reference", "Frontend Dev Handbook"]

SCOPO DEL PROGETTO:
  [Descrivi in 3-5 frasi lo scopo della knowledge base. A chi serve? Quale
   problema risolve? Chi e' il lettore ideale?]
  Esempio: "Questa KB e' un riferimento tecnico per ingegneri che lavorano
  con pipeline di dati. Copre concetti, strumenti e pattern dal dato grezzo
  all'analisi. E' pensata per livelli da junior a senior."

  PROMPT DA FORNIRE ALL'AI
================================================================================

Voglio costruire una knowledge base tecnica con le caratteristiche esatte
descritte di seguito. Genera tutti i file necessari rispettando fedelmente

4. Le convenzioni da rispettare

Non aggiungere funzionalita' non richieste. Non semplificare la struttura.
Genera i file esattamente come specificato.

--------------------------------------------------------------------------------
  5. FRONTMATTER OBBLIGATORIO — Ogni file .md di contenuto
--------------------------------------------------------------------------------

Ogni file .md (esclusi index.md, tags.md, _index.md di categoria) DEVE avere
questo frontmatter YAML esatto in testa. Tutti i campi sono obbligatori:

---
title: "Nome Argomento"
slug: nome-argomento
category: nome-categoria
tags: [tag1, tag2, tag3]
search_keywords: [sinonimo1, sinonimo2, acronimo, termine-alternativo, variante-en, variante-it]
parent: categoria/_index
related: [categoria/slug-argomento-correlato-1, categoria/slug-argomento-correlato-2]
official_docs: https://link-documentazione-ufficiale.com
status: draft
difficulty: intermediate
last_updated: YYYY-MM-DD
---

Spiegazione dei campi:
  title         : Titolo leggibile della pagina
  slug          : Identico al nome file senza estensione, kebab-case
  category      : Categoria principale di appartenenza
  tags          : Elenco tag brevi, lowercase, in inglese, plurale dove sensato
  search_keywords: Sinonimi, acronimi, varianti nella lingua dei contenuti e in
                   inglese. Questo campo potenzia la ricerca semantica: includere
                   tutte le varianti con cui qualcuno potrebbe cercare l'argomento.
  parent        : Percorso relativo al _index.md della sezione padre
  related       : Lista di slug di argomenti correlati (per le sezioni Relazioni)
  official_docs : URL della documentazione ufficiale di riferimento
  status        : draft | in-progress | complete | needs-review
  difficulty    : beginner | intermediate | advanced | expert
  last_updated  : Data dell'ultimo aggiornamento in formato YYYY-MM-DD

REGOLE NAMING:
  - File .md: kebab-case (es. "event-driven-architecture.md")
  - Cartelle: kebab-case (es. "cloud-providers/")
  - Slug: identico al nome file senza estensione
  - Tags: lowercase, brevi, inglese
  - Diagrammi: [categoria]-[argomento]-[descrizione].drawio.svg

--------------------------------------------------------------------------------
  7. STRUTTURA DI UNA PAGINA CONTENUTO — Standard di scrittura
--------------------------------------------------------------------------------

Sezioni standard (in ordine):
  H1  : Titolo (1 solo per file, identico al campo title del frontmatter)
  H2  : ## Panoramica
  H2  : ## Concetti Chiave
  H2  : ## Architettura / Come Funziona
  H2  : ## Configurazione & Pratica
  H2  : ## Best Practices
  H2  : ## Troubleshooting
  H2  : ## Relazioni
  H2  : ## Riferimenti

Regole di scrittura:
  - Un solo H1 per file
  - H2 per le sezioni principali, H3+ per sotto-sezioni
  - Code blocks SEMPRE con language tag (```yaml, ```bash, ```python, ```json, ecc.)
  - Acronimi espansi alla prima comparsa nel testo in prosa:
    es. "HTTP (HyperText Transfer Protocol)" — poi solo "HTTP"
  - Non espandere acronimi nei code block o nei commenti di codice
  - Testo in prosa completo: ogni concetto introdotto deve essere spiegato,
    non dare nulla per scontato anche se l'argomento sembra ovvio

Admonitions disponibili:
  !!! note "Titolo"        <- informazione supplementare
  !!! warning "Titolo"     <- aspetto critico
  !!! tip "Titolo"         <- best practice / consiglio
  !!! example "Titolo"     <- caso d'uso concreto
  !!! info "Titolo"        <- contesto aggiuntivo

  Versione collassabile (per la sezione Relazioni):
  ??? info "Titolo"        <- chiusa di default, clic per aprire
  ???+ info "Titolo"       <- aperta di default

Formattazione tipografica:
  ==testo==      <- evidenziato giallo
  ^^testo^^      <- inserito (underline)
  ~~testo~~      <- cancellato (strikethrough)
  `codice`       <- inline code
  **grassetto**
  *corsivo*

--------------------------------------------------------------------------------
  12. MEGA-MENU JAVASCRIPT — docs/assets/js/nav-enhance.js
--------------------------------------------------------------------------------

Genera docs/assets/js/nav-enhance.js con questo contenuto esatto.
Questo script costruisce automaticamente un mega-menu a discesa sotto ogni
tab di navigazione, con sezioni e sotto-link organizzati in griglia.

---INIZIO FILE nav-enhance.js---

(function () {
  var activeTop = null;
  var hideTimer = null;

  function scheduleHide() {
    clearTimeout(hideTimer);
    hideTimer = setTimeout(hideAll, 250);
  }

  function cancelHide() {
    clearTimeout(hideTimer);
  }

  function hideAll() {
    cancelHide();
    if (activeTop) {
      activeTop.classList.remove('kbdrop--visible');
      activeTop = null;
    }
  }

  function measureWidth(el) {
    el.style.visibility = 'hidden';
    el.style.display    = 'block';
    var w = el.offsetWidth;
    el.style.display    = '';
    el.style.visibility = '';
    return w;
  }

  function showTop(drop, anchorRect) {
    cancelHide();
    if (activeTop === drop) return;
    if (activeTop) activeTop.classList.remove('kbdrop--visible');

    var dw   = measureWidth(drop);
    var left = anchorRect.left + anchorRect.width / 2 - dw / 2;
    left = Math.max(8, Math.min(left, window.innerWidth - dw - 8));

    drop.style.top       = (anchorRect.bottom - 1) + 'px';
    drop.style.left      = left + 'px';
    drop.style.maxHeight = (window.innerHeight - anchorRect.bottom - 12) + 'px';
    drop.classList.add('kbdrop--visible');
    activeTop = drop;
  }

  function normHref(href) {
    return href
      .replace(/_index\/?$/, '')
      .replace(/index\.html$/, '')
      .replace(/\/?$/, '/');
  }

  function findDirectChildren(sectionHref) {
    var base     = normHref(sectionHref);
    var children = [];
    var seen     = {};

    document.querySelectorAll('a.md-nav__link').forEach(function (a) {
      var href = a.href;
      if (!href) return;
      var norm = normHref(href);
      if (seen[norm]) return;
      if (!norm.startsWith(base)) return;
      if (norm === base) return;

      var rel   = norm.slice(base.length).replace(/\/?$/, '/');
      var parts = rel.split('/').filter(Boolean);
      if (parts.length !== 1) return;

      seen[norm] = true;
      var textEl = a.querySelector('.md-ellipsis');
      var text   = (textEl ? textEl.textContent : a.textContent).trim();
      if (text) children.push({ text: text, href: href });
    });

    return children;
  }

  function buildMenu(children) {
    var sections = children.map(function (c) {
      return { text: c.text, href: c.href, subs: findDirectChildren(c.href) };
    });

    var drop = document.createElement('div');
    drop.className = 'kbdrop kbdrop--mega';
    drop.setAttribute('data-kb-drop', '');

    sections.forEach(function (s) {
      var section = document.createElement('div');
      section.className = 'kbdrop__section';

      var header        = document.createElement('a');
      header.className  = 'kbdrop__section-header';
      header.href       = s.href;
      header.textContent = s.text;
      section.appendChild(header);

      if (s.subs.length) {
        var grid = document.createElement('ul');
        grid.className = 'kbdrop__section-items';
        s.subs.forEach(function (sub) {
          var li = document.createElement('li');
          var a  = document.createElement('a');
          a.href        = sub.href;
          a.textContent = sub.text;
          li.appendChild(a);
          grid.appendChild(li);
        });
        section.appendChild(grid);
      }

      drop.appendChild(section);
    });

    return drop;
  }

  function buildDropdowns() {
    document.querySelectorAll('[data-kb-drop]').forEach(function (d) { d.remove(); });
    hideAll();

    var tabsList = document.querySelector('.md-tabs__list');
    if (!tabsList) return;

    var tabItems        = Array.from(tabsList.querySelectorAll(':scope > .md-tabs__item'));
    var rootNames       = ['home', 'tags'];
    var categoryStarted = false;

    tabItems.forEach(function (item) {
      item.classList.remove('md-tabs__item--category-start');
      if (item._kbEnter) item.removeEventListener('mouseenter', item._kbEnter);
      if (item._kbLeave) item.removeEventListener('mouseleave', item._kbLeave);
      item._kbDrop = null;
    });

    tabItems.forEach(function (tabItem) {
      var link = tabItem.querySelector('.md-tabs__link');
      if (!link) return;

      var name   = link.textContent.trim().toLowerCase();
      var isRoot = rootNames.indexOf(name) !== -1;

      if (!isRoot && !categoryStarted) {
        tabItem.classList.add('md-tabs__item--category-start');
        categoryStarted = true;
      }
      if (isRoot) return;

      var children = findDirectChildren(link.href);
      if (!children.length) return;

      var drop = buildMenu(children);
      document.body.appendChild(drop);
      tabItem._kbDrop = drop;

      drop.addEventListener('mouseenter', function () { cancelHide(); activeTop = drop; });
      drop.addEventListener('mouseleave', scheduleHide);

      tabItem._kbEnter = function () {
        cancelHide();
        showTop(drop, tabItem.getBoundingClientRect());
      };
      tabItem._kbLeave = scheduleHide;

      tabItem.addEventListener('mouseenter', tabItem._kbEnter);
      tabItem.addEventListener('mouseleave', tabItem._kbLeave);
    });
  }

  window.addEventListener('scroll', hideAll, { passive: true });
  document.addEventListener('click', function (e) {
    if (!activeTop || !activeTop.contains(e.target)) hideAll();
  });

  if (typeof document$ !== 'undefined') {
    document$.subscribe(buildDropdowns);
  } else {
    window.addEventListener('DOMContentLoaded', buildDropdowns);
  }
})();

--------------------------------------------------------------------------------
  14. SISTEMA DI NAVIGAZIONE — Come funziona
--------------------------------------------------------------------------------

La navigazione e' completamente automatica grazie a tre meccanismi:

A) PLUGIN AWESOME-PAGES
   Genera la sidebar e i tab automaticamente dall'albero delle cartelle.
   Non serve un nav: manuale in mkdocs.yml. L'ordine delle voci segue
   l'ordine alfabetico dei file, modificabile con file .pages nelle cartelle.

B) NAVIGATION.INDEXES
   Ogni file _index.md diventa la landing page della sua cartella.
   Cliccando un tab o una sezione si apre direttamente il suo _index.md.

C) MEGA-MENU (nav-enhance.js)
   Al passaggio del mouse sui tab della navbar, appare un menu a discesa
   con tutte le sotto-sezioni della categoria, organizzate in griglia.
   Le voci "Home" e "Tags" restano a sinistra; le categorie si allineano
   a destra grazie alla classe CSS .md-tabs__item--category-start.
   Il menu si costruisce dinamicamente leggendo i link gia' presenti nel DOM
   (sidebar nascosta che MkDocs genera sempre), quindi non richiede
   configurazione manuale.

FILE .pages (opzionale per controllare l'ordine):
   Crea un file .pages in qualsiasi cartella con questo formato:
   ---
   order:
     - fondamentali
     - argomento-1.md
     - argomento-2.md
   ---


--------------------------------------------------------------------------------
  15. SISTEMA DI RICERCA — Come funziona
--------------------------------------------------------------------------------

MkDocs Material offre ricerca full-text nativa con due livelli:

LIVELLO 1 — RICERCA STANDARD (automatica)
  Indicizza automaticamente tutto il testo di ogni pagina.
  Shortcut: tasto "/" o "S" per aprire la barra di ricerca.
  Funzionalita': suggerimenti in tempo reale, highlight dei risultati,
  URL condivisibile per ogni ricerca.

LIVELLO 2 — RICERCA SEMANTICA AMPLIATA (tramite search_keywords)
  Il campo search_keywords nel frontmatter viene indicizzato insieme al
  contenuto della pagina. Questo permette di trovare una pagina cercando
  sinonimi, acronimi, varianti linguistiche o termini correlati che non
  compaiono nel testo principale.
  Esempio: la pagina "TLS/SSL" include search_keywords come
  ["transport layer security", "https", "certificato", "crittografia",
  "ssl certificate", "tls 1.3", "handshake"] — cosi' e' trovabile con
  qualsiasi di questi termini anche se nel testo appare solo "TLS".

REGOLA per search_keywords:
  Includere sempre: acronimi, forma estesa, sinonimi in italiano e inglese,
  abbreviazioni comuni, termini tecnici alternativi, nomi di tool/librerie
  correlati che un utente potrebbe cercare aspettandosi di trovare questa pagina.


--------------------------------------------------------------------------------
  16. SISTEMA DI RELAZIONI — Come funziona
--------------------------------------------------------------------------------

Le relazioni tra argomenti sono gestite in due modi complementari:

A) FRONTMATTER related:
   Lista di slug di argomenti direttamente correlati.
   Serve per la scoperta programmatica delle relazioni e per eventuali
   estensioni future (grafo delle relazioni, pagina "vedi anche", ecc.).
   Formato: percorso relativo dalla root docs senza estensione .md
   Esempio: related: [sicurezza/tls, protocolli/http2, fondamentali/dns]

B) SEZIONE ## Relazioni nel testo:
   Usa admonitions collassabili per link contestuali ad argomenti correlati.
   Ogni admonition include 2-3 frasi di contesto + link all'approfondimento.
   Formato:

   ??? info "Nome Argomento Correlato — Contesto"
       Breve spiegazione del perche' questo argomento e' rilevante qui.
       2-3 frasi che contestualizzano la relazione.

       **Approfondimento completo -->** [Nome Argomento](../percorso/argomento.md)

PRINCIPIO DI ISOLAMENTO:
   Ogni file .md e' un'unita' autonoma. Aggiungere o modificare un argomento
   non richiede MAI di modificare altri file (a meno di non voler aggiornare
   esplicitamente i related di altre pagine). Il costo di ogni operazione e'
   O(1) rispetto alla dimensione del progetto.


--------------------------------------------------------------------------------
  17. DIAGRAMMI — Come funziona
--------------------------------------------------------------------------------

Per diagrammi complessi usa file .drawio.svg salvati in docs/assets/diagrams/.

NAMING: [categoria]-[argomento]-[descrizione].drawio.svg
  Esempio: networking-tls-handshake-flow.drawio.svg

EMBED nella pagina:
  <div class="diagram-container">
    <img src="../../assets/diagrams/[nome-file].drawio.svg"
         alt="Descrizione del diagramma" />
  </div>

Il file .drawio.svg e' indipendente dal .md:
  - Modifiche al .md non richiedono rigenerazione del diagramma
  - Rigenerazione del diagramma non richiede modifica del .md
  - Il path non cambia mai

Per diagrammi semplici usa ASCII art direttamente nel testo:
  ```
  Client --> Server --> Database
  ```

Per diagrammi con Mermaid (supportato via pymdownx.superfences):
  ```mermaid
  graph LR
    A[Client] --> B[Server]
    B --> C[Database]
  ```

--------------------------------------------------------------------------------
  20. PRINCIPI OPERATIVI — Istruzioni per chi mantiene la KB
--------------------------------------------------------------------------------

AGGIUNGERE UN NUOVO ARGOMENTO:
  1. Identificare la categoria corretta
  2. Verificare che l'argomento non esista gia' (controllare nomi file)
  3. Creare il file .md con frontmatter completo dal TEMPLATE.md
  4. Compilare TUTTI i campi (nessuno opzionale)
  5. Popolare search_keywords con tutti i sinonimi rilevanti
  6. Popolare related con gli slug degli argomenti collegati
  7. Non modificare altri file

MODIFICARE UN ARGOMENTO ESISTENTE:
  1. Leggere il file dell'argomento
  2. Applicare la modifica
  3. Aggiornare last_updated
  4. Se la modifica introduce nuove relazioni, aggiornare related nello stesso file
  5. Non leggere ne' modificare altri file

COLLEGARE DUE ARGOMENTI (aggiungere relazione):
  1. Aprire file A -> aggiungere B nel campo related
  2. Aprire file B -> aggiungere A nel campo related
  3. Aggiornare last_updated in entrambi
  (Unico caso in cui si toccano 2 file — costo O(2), mai di piu')

QUALITA' DEI CONTENUTI:
  - Ogni acronimo espanso alla prima comparsa nel testo
  - Code block sempre con language tag
  - Sezione Panoramica completa: cos'e', perche' esiste, quando usarlo,
    quando NON usarlo
  - Sezione Troubleshooting con i problemi reali piu' comuni
  - Nessun contenuto incompleto o placeholder nel testo finale
